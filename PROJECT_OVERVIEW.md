# PitchIQ ‚Äî Project Overview

> A detailed snapshot of the PitchIQ codebase, architecture, and current state.

## 1. What it is

A football intelligence web app that turns two team names into a structured pre-match tactical report. The report is generated by Google Gemini (`gemini-2.5-flash`), grounded against the bundled FIFA World Cup 2026 dataset, and rendered into a React dashboard.

**Tagline:** *Football Intelligence, Reimagined.*

**Flagship demo:** four pre-defined matchups ‚Äî Argentina vs France, Brazil vs Germany, England vs Portugal, Japan vs Morocco.

### Request flow for `POST /api/analyze`

1. FastAPI route (thin controller) validates `AnalysisRequest` with Pydantic (`homeTeam`, `awayTeam`, distinct, names 2‚Äì80 chars).
2. `AnalysisService.analyze()`:
   - Loads `tactical_analysis_system.md` prompt.
   - Calls `FootballContextBuilder.build(home, away)` to assemble a grounded knowledge block from `KnowledgeService` (only the two teams' data ‚Äî no leak).
   - Renders `tactical_analysis_user.md` with `$home_team`, `$away_team`, `$football_context`.
   - Calls `GeminiService.generate_json(...)` with `response_json_schema` enforced by Gemini.
   - Validates the response against `AnalysisResponse` (Pydantic) **plus** a `_validate_match_identity` invariant: `homeTeam.name` and `awayTeam.name` must match the request, `predictedWinner ‚àà {home, away, "Draw"}`.
3. On validation failure ‚Üí re-issues `tactical_analysis_repair.md` with `$validation_error` and `$invalid_response` injected. Up to `GEMINI_MAX_VALIDATION_ATTEMPTS` (default 2). If all attempts fail ‚Üí `502 invalid_analysis_response`.
4. Wraps in `AnalysisApiResponse { data, model }` and returns JSON.

## 5. Tech stack

- **Frontend:** React 19.1, Vite 6.3, TypeScript 5.8 strict, TanStack Query 5.81, React Router 7, Tailwind CSS v4.
- **Backend:** Python 3.12, FastAPI 0.115, Pydantic 2, pydantic-settings 2.10, Uvicorn 0.34.

- **Knowledge:** Bundled FIFA World Cup 2026 JSON parsed once into frozen dataclasses (`TeamRecord`, `GroupRecord`, `StadiumRecord`, `PlayerRecord`, `SquadRecord`, `FixtureRecord`), cached via `@lru_cache`.
- **Infra:** Docker, Docker Compose, Firebase Hosting, Google Cloud Run, Google Secret Manager.

## 6. Knowledge layer

`KnowledgeService` loads six JSON files once at startup (`backend/knowledge/`), normalises them into typed records, and exposes:

- `resolve_team(name)` ‚Äî case-insensitive, supports an alias dict (USA, South Korea, Czechia, T√ºrkiye, IR Iran, DR Congo, C√¥te d'Ivoire, Cabo Verde) and a substring fuzzy fallback.
- `get_team`, `search_team`, `get_group` (accepts "A" or "Group A"), `get_squad`, `get_fixture(home, away)` (ordered, paired on either side), `get_stadium` (with city-token + substring fuzzy fallback), `all_fixtures`.
- Alias dict + canonical resolution order is carefully documented so that downstream squad lookups keep working when canonical name differs from the dataset raw label.
- Public counters: `team_count`, `group_count`, `stadium_count`, `squad_count`, `fixture_count` are logged at startup.

`FootballContextBuilder` queries **only** the two requested teams' slice ‚Äî squads (top 3 per position GK/DF/MF/FW), group opponents, scheduled fixtures between them, and the venue ‚Äî and emits a deterministic Markdown knowledge block injected into the prompt.

## 7. Prompt engineering

Three versioned Markdown templates under `backend/app/prompts/`:

- **`tactical_analysis_system.md`** ‚Äî Casts Gemini as a UEFA Pro Licensed Tactical Analyst, defines the JSON-only output contract, forbids inventing injuries/lineups/transfers/live facts, mandates exact team names and 0‚Äì100 scores.
- **`tactical_analysis_user.md`** ‚Äî Interpolates `$home_team`, `$away_team`, `$football_context`. Asks for possession / out-of-possession / transitions / game-state analysis.
- **`tactical_analysis_repair.md`** ‚Äî Retry prompt interpolating `$validation_error` and `$invalid_response` so the model can self-correct.

`PromptLoader` uses Python `string.Template` for safe variable substitution.

### In-process analysis cache

`AnalysisService` memoises successful analysis responses in-process using `functools.lru_cache`. Identical `(homeTeam, awayTeam)` requests skip Gemini I/O and return from memory in milliseconds.

- Toggle: `ANALYSIS_CACHE_ENABLED` (`Settings.analysis_cache_enabled`, default `True`). When disabled, every request hits Gemini.
- Capacity: `ANALYSIS_CACHE_SIZE` (`Settings.analysis_cache_size`, default `128`; range `1‚Äì1024`). Used directly as `lru_cache(maxsize=...)`.
- Observability: `AnalysisService.cache_info()` returns `hits`, `misses`, `maxsize`, `currsize` for tests and runtime introspection; exercised by `backend/tests/test_analysis_service.py`.
- Scope: per-process; not shared across Cloud Run instances and not persisted. Cache keys are the canonical team-name pair from `KnowledgeService`.

## 8. Response schema (Pydantic, `app/schemas/analysis.py`)

Top-level `AnalysisResponse` (camelCase via `to_camel` alias generator, `extra="forbid"`, `populate_by_name=True`):

| Field | Type | Constraint |
| --- | --- | --- |
| `matchOverview` | string | 12‚Äì700 chars |
| `homeTeam` / `awayTeam` | `TeamProfile` | name + formation (2‚Äì160) + playingStyle (12‚Äì700) + tacticalIdentity (12‚Äì700) |
| `predictedWinner` | string | exact home name, exact away name, or `"Draw"` |
| `confidence` | int | 0‚Äì100 |
| `formations` | `FormationMatchup` | home + away + matchup narrative |
| `strengths` / `weaknesses` | `TeamFactors` | 2‚Äì4 entries per side, 12‚Äì700 chars each |
| `keyBattles` | array | 3‚Äì5 entries, each with `zone` + `homePlayerOrUnit` + `awayPlayerOrUnit` + `edge ‚àà {home, away, even}` + `analysis` |
| `tacticalInsights` | array | 3‚Äì6 entries, `importance ‚àà {critical, high, medium}` |
| `expectedGameFlow` | `GameFlow` | opening / middle / closing phases |
| `coachRecommendation` | object | home + away + `decisiveAdjustment` |
| `pressingScore`, `possessionScore`, `attackingThreat`, `defensiveStability` | `TeamScore` | home + away, each 0‚Äì100 |

Post-validation invariants (enforced in `AnalysisService._validate_match_identity`): `homeTeam.name` and `awayTeam.name` must match the request (case-insensitive); `predictedWinner` must be one of the two team names or `"Draw"`. Mismatches trigger the repair path.

> The router (`backend/app/api/router.py`) mounts the analysis endpoints at the `/api` prefix and the health endpoint at the root. The frontend nginx container additionally serves a static `/healthz` for its own platform probe ‚Äî that route is owned by the static site, not the API.

| Method | Path | Status |
| --- | --- | --- |
| `GET` | `/health` | Live |
| `POST` | `/api/analyze` | Live, returns `AnalysisApiResponse` |

### Error contract

All errors return `{ "error": { "code": "...", "message": "..." } }`:

| Status | Code | Trigger |
| --- | --- | --- |
| 422 | `invalid_request` | Pydantic body validation failed (identical team names, missing fields) |
| 502 | `invalid_analysis_response` | Model failed validation on every attempt |
| 502 | `gemini_unavailable` | Provider returned error or empty body |
| 503 | `gemini_not_configured` | `GEMINI_API_KEY` is empty |

`core/errors.py` defines `AppError` plus `GeminiConfigurationError`, `GeminiProviderError`, `AnalysisGenerationError`. `main.py` registers exception handlers that map them to the correct HTTP status.

## 10. Frontend surface

- **Routes:** `/` (Home), `/analysis` (Match Analysis ‚Äî flagship), `*` (404).
- **State:** TanStack Query `useMutation` per request, with an `AbortController` ref for cancellation on resubmit and on unmount.
- **Networking:** `requestMatchAnalysis` in `services/analysisService.ts` resolves `VITE_API_BASE_URL + /api/analyze`, has its own 30 s timeout and merges an external `AbortSignal`, throws a typed `AnalysisApiError` carrying `status`/`code`/`message`/`details`.
- **Match analysis page:** form ‚Üí loading state (with team names echoed) ‚Üí error state with retry ‚Üí results dashboard. Results are broken into components: `MatchHeader`, `ScoreBar`, `FormationsSection`, `FactorsList`, `KeyBattles`, `GameFlow`, `CoachRecommendation`, `TeamPanel`, plus loading/error states.
- **Types:** `frontend/src/types/analysis.ts` mirrors the Pydantic schema 1:1 (camelCase), with `Importance`, `Edge`, `Score` literal unions.
- **Accessibility:** semantic HTML, visible focus, sufficient contrast, accessible loading/error/empty states per AGENTS.md.

## 13. Deployment

> **See:** [README.md ‚Üí Deployment](README.md#deployment) for complete Cloud Run and Firebase Hosting commands, secret management, and **critical post-deployment verification steps**.

**Backend ‚Äî Cloud Run:**

```bash
gcloud run deploy pitchiq-api \
  --source backend \
  --region YOUR_REGION \
  --allow-unauthenticated \
  --min-instances 1 \
  --max-instances 5 \
  --concurrency 10 \
  --timeout 120 \
  --set-env-vars APP_ENV=production,CORS_ORIGINS=https://pitchiq-ai.web.app \
  --set-secrets GEMINI_API_KEY=GEMINI_API_KEY:latest
```

> **Sizing rationale.** `--min-instances 1` keeps one container warm to eliminate cold starts (4‚Äì8 s import + knowledge init). `--max-instances 5` caps cost under burst. `--concurrency 10` bounds per-instance queue depth because each analysis holds the event loop ~10‚Äì40 s in Gemini I/O. `--timeout 120` is above the worst-case `2 √ó 45 s` orchestration path but well below the 300 s default, so stuck requests fail fast. CPU allocation, memory, vCPU count, and Uvicorn workers are left at defaults; revisit after measuring real traffic.

**Frontend ‚Äî Firebase Hosting:**

```bash
npm --prefix frontend run build
firebase use --add
firebase deploy --only hosting
```

> **Important:** After deployment, perform the end-to-end verification checks documented in the README to ensure CORS is configured correctly and the full pipeline works in production.

## 14. Engineering guardrails (`AGENTS.md`)

- `frontend/` and `backend/` deploy independently.
- FastAPI never serves frontend assets.
- API origins and secrets in env vars only; `.env` is gitignored.
- No auth, databases, queues, or new endpoints without explicit approval.
- Routes are thin HTTP controllers; services own use-case orchestration; the Gemini client owns provider calls; prompt modules own prompt text; Pydantic schemas validate every model response.
- Strict TypeScript; no `any` to bypass design. Type all Python function signatures and module boundaries.
- Components must include loading, empty, and error states.
- Conventional Commits; focused commits only.
- Definition of done: contracts typed, errors handled, tests pass, docs/env examples current, secrets external, deployment assumptions still hold.

## 15. What's intentionally left for later

- **User Interface Evolution:** The current UI is functional, clean, and suitable for demonstrating the product.
